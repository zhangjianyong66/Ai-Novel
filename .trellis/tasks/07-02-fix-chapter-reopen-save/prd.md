# 修复定稿章节回退草稿后保存失败

## Goal

用户在写作页把已定稿章节切换为草稿后，点击“保存”应成功把章节回退到起草中，而不是因为前端携带完整章节字段触发后端 `chapter_done_readonly` 校验失败。

## Background

- 用户复现路径：前端写作页将已定稿章节切换为草稿后点击“保存”。
- 保存按钮当前会提交完整章节表单字段。
- 后端约定：已定稿章节只允许请求体单独把 `status` 改为 `drafting`；如果同时提交正文、标题、摘要等字段，会返回 `VALIDATION_ERROR`，`details.reason=chapter_done_readonly`。
- 该后端约定已记录在项目 `AGENTS.md` 的“章节自动更新约定”中，应保持不变。

## Requirements

- 修复写作页保存流程，使“已定稿 -> 草稿”的保存请求满足后端只允许单字段回退的契约。
- 用户只切换草稿且没有编辑内容时，点击保存应成功，并刷新前端基线状态，避免继续显示未保存。
- 不放宽后端对已定稿章节内容编辑的只读保护。
- 不改变普通草稿章节保存、定稿章节内容只读提示、一键保存并触发更新等其他行为。

## Acceptance Criteria

- [x] 已定稿章节在前端切换为草稿后，点击“保存”不再返回 `chapter_done_readonly`。
- [x] 回退草稿保存成功后，前端章节状态为 `drafting`，dirty 状态清除。
- [x] 已定稿章节若请求体同时修改内容字段，后端仍返回 `chapter_done_readonly`。
- [x] 添加或更新覆盖该回归的前端测试；必要时保留后端现有测试覆盖只读契约。
- [x] 运行相关前端/后端测试命令并记录结果。

## Out of Scope

- 不新增独立“回退草稿”后端接口。
- 不允许已定稿章节通过完整保存直接修改正文、标题、计划或摘要。
- 不调整章节自动更新触发语义。
