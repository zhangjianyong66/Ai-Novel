# 用户管理支持管理员设置与用户名修改

## Goal

让管理员能在用户管理页维护现有用户的管理员权限，并能把 Linux.do 登录自动生成的固定格式登录标识改成可读、可自定义的登录用户名。

用户价值：
- Linux.do OIDC 新用户当前会自动生成 `linuxdo_<login>` 格式的 `users.id`，用户无法在注册阶段自定义。
- 管理员需要后续把该登录用户名改成更合适的值，同时尽量降低对既有业务外键和 Linux.do 绑定的影响。
- 管理员需要能对已存在用户授予或撤销管理员权限，而不仅是在创建用户时设置。

## Background

- 后端用户主表是 `users`，主键 `User.id` 当前也是本地登录使用的 `user_id`，长度 64。
- `User` 已有 `is_admin`、`display_name`、`email` 字段。
- Linux.do OIDC 首次登录会通过 `_linuxdo_suggest_user_id()` 生成 `linuxdo_<login>` 风格的 `User.id`，并通过 `auth_external_accounts.user_id` 绑定外部账号。
- 当前管理员接口已有：
  - `GET /api/auth/admin/users`：列表、筛选、统计。
  - `POST /api/auth/admin/users`：创建用户，并可设置 `is_admin`。
  - `POST /api/auth/admin/users/{target_user_id}/password/reset`：管理员重置密码。
  - `POST /api/auth/admin/users/{target_user_id}/disable`：启用/禁用本地密码登录账号。
- 当前前端 `AdminUsersPage` 已有创建用户、搜索分页、重置密码、启用/禁用操作，但用户列表没有修改用户名或修改管理员状态的操作。
- 多张业务表通过 `owner_user_id`、`actor_user_id`、成员关系等字段引用 `users.id`；直接修改主键影响面大，需要同步所有外键与关联记录。

## Decisions

- 新增可修改、唯一的本地登录用户名字段 `users.login_name`；`users.id` 保持为稳定内部用户 ID，不再作为可修改登录名。
- 本地登录表单中的“用户名”以后匹配 `login_name`；为兼容现有数据，迁移时应给已有用户补 `login_name = users.id`。
- API 响应继续返回稳定内部 `id`，同时新增 `login_name`；前端用户管理页突出展示和编辑 `login_name`，内部 `id` 以弱化信息展示，便于排障。
- 本次纳入管理员编辑用户基础资料，支持修改 `login_name`、`display_name`、`email`；管理员权限和禁用仍作为危险操作单独处理。
- 修改 `login_name` 后，旧登录名立即失效，不提供旧登录名别名兼容。
- 修改 `login_name` 或管理员权限后，不强制踢出现有会话；权限以每次后端鉴权实时查询为准，前端在刷新当前用户信息后同步状态。
- 本地登录、本地注册、管理员创建用户等面向“登录用户名”的请求字段直接改为 `login_name`，不继续沿用 `user_id` 作为登录名入参。
- 旧的 `user_id` 登录名入参直接拒绝，不做兼容别名。
- 新增本地用户时，`users.id` 使用系统生成的稳定随机 ID，`login_name` 单独保存登录用户名；迁移历史用户时保留原 `users.id`，并补 `login_name = 原 users.id`。
- 管理员不能撤销自己的管理员权限。
- `admin` 作为超级管理员，不允许被撤销超级管理员/管理员权限。
- `admin` 超级管理员禁止禁用、禁止修改登录用户名，只允许改显示名、邮箱、密码。
- 登录用户名只允许小写字母 `a-z`、数字 `0-9`、下划线 `_`、短横线 `-`，长度 1 到 64；输入应统一归一化为小写；中文名等展示需求使用 `display_name`。

## Requirements

- 管理员可以在用户管理页修改现有用户的登录用户名。
- 管理员可以在用户管理页修改现有用户的显示名和邮箱。
- 登录用户名应优先设计为独立字段，`users.id` 保持稳定内部用户 ID。
- 新建本地用户的内部 `users.id` 不再等于 `login_name`，应由系统生成。
- 历史用户迁移时不得修改 `users.id`。
- 用户相关 API 应保留稳定内部 `id` 字段，并新增 `login_name` 字段。
- 面向登录用户名的请求字段应使用 `login_name`，包括本地登录、本地注册和管理员创建用户。
- 旧的 `user_id` 登录名入参应按请求模型校验失败处理，不兼容映射到 `login_name`。
- 用户管理页应以“登录用户名”为主展示字段，内部 ID 可作为辅助信息展示。
- 修改登录用户名后，用户仍能用新用户名进行本地登录；旧登录用户名不再可用于本地登录。
- 修改登录用户名不提供旧用户名别名或兼容期。
- 修改登录用户名不应破坏 Linux.do OIDC 外部账号绑定。
- 登录用户名必须唯一，且按统一小写后的结果判重。
- 登录用户名不得包含中文、空格或大写保留；大写输入应归一化为小写。
- 修改 Linux.do OIDC 用户名后，原 Linux.do 外部账号绑定仍应指向同一个用户，之后继续用 Linux.do 登录应进入改名后的账号。
- 管理员可以在用户管理页授予或撤销现有用户的管理员权限。
- 管理员不能撤销自己的管理员权限。
- 超级管理员 `admin` 不允许被撤销管理员权限。
- 超级管理员 `admin` 不允许被禁用。
- 超级管理员 `admin` 不允许修改登录用户名。
- 超级管理员 `admin` 允许修改显示名、邮箱和密码。
- 管理员权限变更和禁用/启用属于危险操作，应与基础资料编辑分开交互并二次确认。
- 修改登录名、管理员权限或禁用状态不新增全局会话吊销机制。
- 管理员列表、统计和当前登录用户信息应在变更后保持一致。

## Acceptance Criteria

- [ ] 管理员可对列表中的用户执行“修改登录用户名”，成功后列表展示新登录用户名，旧登录用户名不再可用于本地登录。
- [ ] 修改登录用户名后，使用旧登录用户名进行本地登录会失败。
- [ ] 管理员可编辑用户显示名和邮箱，成功后列表立即反映变更。
- [ ] 用户管理页能区分展示“登录用户名”和“内部 ID”，避免把 `id` 误认为可修改登录名。
- [ ] 现有本地用户迁移后仍可使用原用户名登录，因为初始 `login_name` 等于原 `users.id`。
- [ ] 新注册用户和管理员新建用户拥有系统生成的稳定内部 `id`，且 `login_name` 为提交的登录用户名。
- [ ] Linux.do 用户修改 `login_name` 后，`users.id` 和 `auth_external_accounts.user_id` 不变。
- [ ] 本地登录、本地注册、管理员创建用户使用 `login_name` 作为请求字段；前端不再把登录用户名提交为 `user_id`。
- [ ] 本地登录、本地注册、管理员创建用户提交旧字段 `user_id` 且缺少 `login_name` 时，请求被拒绝。
- [ ] 修改登录用户名时如果包含非法字符、为空、超长或归一化后已存在，后端返回明确错误，不产生部分更新。
- [ ] 修改 Linux.do 用户登录用户名后，再次 Linux.do 登录不会创建重复用户，并仍进入原账号。
- [ ] 管理员可对现有用户执行“设为管理员/撤销管理员”，成功后列表和统计中的管理员数量更新。
- [ ] 当前管理员撤销自己的管理员权限时，后端拒绝并返回明确错误。
- [ ] 对超级管理员 `admin` 执行撤销管理员权限时，后端拒绝并返回明确错误。
- [ ] 对超级管理员 `admin` 执行禁用或修改登录用户名时，后端拒绝并返回明确错误。
- [ ] 超级管理员 `admin` 仍可通过既有密码修改/重置能力维护密码，并可修改显示名/邮箱。
- [ ] 权限变更接口仅管理员可调用。
- [ ] 撤销某用户管理员权限后，该用户后续调用管理员接口应立即被后端拒绝，即使原会话仍存在。
- [ ] 前端对危险操作提供确认，失败时显示后端错误信息。

## Notes

- 该任务会涉及认证、权限和主键改名的兼容性，规划阶段需要补充 `design.md` 与 `implement.md` 后再启动实现。
