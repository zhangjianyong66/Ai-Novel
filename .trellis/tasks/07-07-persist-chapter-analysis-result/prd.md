# 持久化章节分析结果

## Goal

写作页“章节分析”弹窗的最近一次成功分析结果必须可恢复：用户刷新页面或重新进入当前章节后，点击章节分析弹窗能看到上次分析结果，并能明确知道该结果是否仍匹配当前已保存章节内容。

## Background

- 当前手动 `POST /api/chapters/{chapter_id}/analyze` 只把结构化分析结果返回给前端弹窗，前端保存在 React state，刷新或切章后丢失。
- LLM 调用记录会写入 `generation_runs`，但弹窗无法从中恢复可操作的结构化 `analysis`。
- 当前 `POST /api/chapters/{chapter_id}/analysis/apply` 会调用 `apply_chapter_analysis`，把分析写入 `plot_analysis.analysis_json` 并重建该章节托管的 `StoryMemory`。
- 后台 `plot_auto_update` 已经会在章节定稿后自动执行分析并应用到剧情记忆。

## Requirements

- 每章只保留最近一次解析成功的章节分析结果，不做历史版本 UI。
- 有未保存修改时不能执行章节分析：前端禁用分析按钮并提示原因，后端拒绝携带草稿字段的分析请求。
- 手动章节分析必须基于已保存章节内容生成，并在成功后持久化到 `plot_analysis`。
- 持久化记录必须绑定当前已保存章节内容，可通过内容 hash 和/或 `active_version_id` 判断是否过期。
- 正文后续变化后，旧分析可以展示但必须标记过期。
- 过期分析只允许查看，不允许“按建议重写”或“重试应用记忆库”。
- 只持久化解析成功的分析结果；解析失败仍只通过当次响应展示，并保留在 `generation_runs` 供排查。
- 分析成功后自动应用到剧情记忆，页面不再显示常驻“应用到记忆库”按钮。
- 自动应用成功后不自动跳转，使用 toast 提供“查看剧情记忆”的入口。
- 自动应用失败时必须保留并展示已保存分析结果，同时显示失败状态和“重试应用”恢复入口。
- 如果分析成功但提取不到剧情记忆，视为成功空结果，不应报错。
- 重新分析同一章节时覆盖该章旧的自动剧情记忆，保留手动创建的记忆。
- 恢复页面数据时不自动打开弹窗；用户点击章节分析后加载最近结果。
- 复用并扩展 `plot_analysis` 表，不新建平行历史表。

## Acceptance Criteria

- [ ] 写作页无未保存修改时点击“开始分析”，分析成功后刷新页面，再打开章节分析弹窗可看到上次分析内容。
- [ ] 写作页存在未保存修改时，章节分析按钮不可执行；绕过前端发送草稿字段到后端会得到明确错误。
- [ ] 分析成功后自动写入/更新 `plot_analysis`，并自动应用到剧情记忆；常驻“应用到记忆库”按钮不再出现。
- [ ] 自动应用失败时，刷新后仍能恢复分析结果，并显示失败状态与“重试应用”入口。
- [ ] 0 条剧情记忆提取结果不会导致分析失败，UI 显示“未提取到可写入的剧情记忆”语义。
- [ ] 修改并保存章节正文后，重新打开弹窗可看到旧分析被标记为过期，重写和重试应用操作不可用。
- [ ] 解析失败的分析不会覆盖已有持久化分析结果。
- [ ] 后台 `plot_auto_update` 继续可写入 `plot_analysis` 和剧情记忆，不被本次改动破坏。

## Out Of Scope

- 多版本章节分析历史、历史列表、版本命名和清理策略。
- 只分析不入库的用户开关。
- 从 `generation_runs` 恢复任意历史分析结果。
