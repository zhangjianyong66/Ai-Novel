# 前端移动端兼容优化设计

## Scope

首轮按“核心创作路径优先”实施，覆盖：

- 全局应用框架：`AppShell`、移动端导航抽屉、页面主容器。
- 核心创作路径：首页、项目向导/设置入口、大纲页、写作页、阅读/预览页。
- 核心叠层：写作页常用抽屉/弹窗、章节列表抽屉、AI 生成、章节分析、版本、历史、上下文、底部向导条/浮动生成卡片。
- 宽表格和高级页：只处理会导致 body 横向溢出的容器级问题，不做完整移动信息架构重排。

## Technical Approach

1. 先建立移动端可用性基线
   - 使用浏览器或静态审查定位 375px 宽度下的横向溢出源。
   - 优先检查固定宽度、`min-w-*`、`xl:min-w-*`、`w-[...]`、`h-[calc(100vh-...)]`、`fixed bottom-*` 和按钮工具条。

2. 保持桌面布局不回退
   - 桌面断点和多栏结构尽量保留。
   - 移动端只在 base/sm/md/lg 断点上调整排列、宽度、滚动和换行。

3. 优先局部修复，谨慎新增通用样式
   - 使用现有 `btn`、`input`、`textarea`、`select`、`panel`、`ui-focus-ring` 等样式体系。
   - 若多个页面重复需要“手机全宽按钮组”“安全区底部 padding”“局部横向滚动表格容器”，再抽成通用 class 或小组件。

4. 叠层按调用方收敛
   - `Drawer` / `Modal` 已有基础滚动能力，优先修正调用方 `panelClassName`。
   - 手机端抽屉应 `w-full` 或 `max-w-[calc(100vw-...)]`，并保留 `overflow-y-auto` / `min-h-0`。
   - 底部抽屉和浮动卡片需考虑 `env(safe-area-inset-bottom)`。

5. 写作页作为主风险区
   - 工具条在手机端按组换行，选择框和关键按钮不能把容器撑宽。
   - 编辑区顶部动作可降级为多行按钮区，低频动作可保留在更多菜单或允许换行。
   - 章节列表在移动抽屉内保持可滚动，高度不能因头部占位导致底部内容不可达。
   - 流式生成浮动卡片和向导条不能永久遮挡正文编辑。

## Compatibility

- 不修改后端 API 和数据模型。
- 不改变路由结构。
- 不引入新的 UI 框架或浏览器 DOM 测试依赖。
- 保持现有 Tailwind 主题 token 和 `frontend/src/index.css` 组件类。

## Validation Strategy

- 静态检查：搜索固定宽度和溢出风险 class，逐项确认核心路径是否需要调整。
- 自动检查：`cd frontend && npm run lint`。
- 构建检查：必要时运行 `cd frontend && npm run build`。
- 交互检查：启动前端后在 375px 视口检查核心路径，重点确认 body 无横向滚动、按钮可触达、抽屉/弹窗可关闭。

## Rollback Notes

- 优先以小范围 class 调整实现，方便逐文件回滚。
- 若新增通用响应式 class 导致桌面回退，应先回滚通用 class，再恢复为局部修复。
