# 优化章节状态工作流交互

## Goal

优化写作页章节详情区的状态交互，把当前分散的“状态标签、状态按钮、保存、删除、记忆更新提示”收敛为可理解、可执行的章节工作流。用户应能清楚判断当前章节处于“计划中 / 草稿 / 定稿”的哪一步、下一步应该做什么、哪些操作会保存内容、哪些操作会触发长期记忆更新。

第一版聚焦章节详情页右上区域、章节状态保存规则、定稿只读、记忆更新入口、未保存切章保护和删除入口降级，不重构批量生成、生成记录、任务中心和全局工具栏。

## Confirmed Facts

- 章节状态枚举为 `planned`、`drafting`、`done`，前端类型定义在 `frontend/src/types.ts`，后端 schema 定义在 `backend/app/schemas/chapters.py`。
- 现有状态流转白名单在 `backend/app/api/routes/chapters.py`：`planned -> drafting`、`drafting -> planned`、`drafting -> done`、`done -> drafting`。
- 后端保存接口 `PUT /api/chapters/{chapter_id}` 当前禁止携带 `status`，状态必须通过 `PATCH /api/chapters/{chapter_id}/status` 修改。
- 后端保存接口当前对 `done` 章节只读保护：定稿章节有任何内容字段修改会返回 `chapter_done_readonly`。
- 前端写作页状态动作当前由 `frontend/src/pages/writing/writingPageModels.ts` 的 `getChapterStatusActions` 提供，仍使用“开始起草 / 标记为已规划 / 标记为定稿 / 回退为起草中”的旧按钮模型。
- 写作页详情区当前在 `frontend/src/pages/writing/WritingPageSections.tsx` 中把章节工具、删除、保存、触发更新、状态按钮混排在同一区域。
- 前端已有未保存切换章节确认，位于 `frontend/src/pages/writing/useChapterEditor.ts` 的 `requestSelectChapter`。
- 前端已有 `done` 只读基础，位于 `frontend/src/pages/writing/useWritingPageState.ts` 的 `isDoneReadonly` 和详情区输入组件 `readOnly`。
- 当前“记忆更新”入口包括全局工具栏 `onOpenMemoryUpdate` 和详情区“一键保存并触发更新”，后者由 `saveAndTriggerAutoUpdates` 调用 `/api/chapters/{chapter_id}/trigger_auto_updates`。
- 任务中心和后端自动更新任务依赖 `done` 章节；第一版不重构批量生成、任务中心、生成记录和全局工具栏。

## Requirements

### R1. 状态语义

- `计划中`：目录或大纲已经规划好，正文尚未正式生成或编写完成。
- `草稿`：已有正文，正在生成、编写、改写或人工调整。
- `定稿`：正文已确认，默认只读，可进入长期记忆更新流程。
- 主要用户路径为 `计划中 -> 草稿 -> 定稿`。
- `计划中 -> 草稿` 由保存非空正文自动触发。
- `草稿 -> 定稿` 必须由用户明确手动触发。
- `定稿 -> 草稿` 必须由用户明确手动触发，并需要确认。
- `计划中` 不允许直接定稿；必须先保存非空正文进入 `草稿`，再手动定稿。

### R2. 工作流面板

- 章节详情右上区域应拆成“章节工具”和“状态工作流”两个视觉分区。
- 状态工作流应展示：
  - 写作状态：`计划中 / 草稿 / 定稿`
  - 记忆状态：`不可更新 / 待更新 / 已更新 / 可能过期 / 更新失败 / 更新中`
  - 当前主动作
  - 次要动作
  - 更多菜单
- 每个写作状态只突出一个推荐主动作，避免多个状态按钮并列。
- 按钮文案使用业务动作，不直接暴露内部枚举：`保存计划`、`保存为草稿`、`保存并定稿`、`标记为定稿`、`仅保存草稿`、`退回草稿`、`更新记忆`、`重新更新记忆`。

### R3. 保存与自动转草稿

- `计划中` 状态下保存标题、要点、摘要且正文为空时，状态保持 `计划中`。
- `计划中` 状态下保存正文且正文去空白后非空时，状态自动变为 `草稿`。
- 生成正文成功落入编辑器后仍是未保存内容；保存非空正文时自动变为 `草稿`。
- 正文被清空后保存不应自动退回 `计划中`，需要用户从更多菜单手动调整。
- `草稿` 状态下有未保存修改时，主动作显示 `保存并定稿`，次动作显示 `仅保存草稿`。
- `草稿` 状态下无未保存修改时，主动作可显示 `标记为定稿`。

### R4. 定稿只读与记忆更新

- `定稿` 章节默认只读，编辑正文、标题、要点、摘要前必须先 `退回草稿`。
- 记忆更新只对 `定稿` 章节开放，并作为定稿后的主动作。
- 非定稿章节不显示或禁用记忆更新主动作。
- 首次更新记忆可直接执行；重新更新记忆应明确表达这是再次触发更新。
- `定稿 -> 草稿` 后保留已有记忆，不删除旧记忆。第一版仅在有可靠数据时展示 `可能过期`，否则不伪造持久记忆状态。

### R5. 章节工具与危险操作

- `分析`、`标注回溯` 属于章节工具，不属于状态工作流。
- `删除` 不再作为右上角常驻按钮，放入更多菜单底部危险项。
- 删除章节必须二次确认，确认文案应包含当前章节上下文。
- 如果章节有未保存修改，删除前应先经过已有的保存 / 放弃 / 取消保护，避免误删或误丢内容。

### R6. 未保存与处理中状态

- 状态工作流面板必须显式显示未保存状态。
- 切换章节、切换大纲等离开当前章节的场景必须保留未保存保护：保存并继续 / 放弃修改 / 取消。
- 保存、定稿、退回草稿、记忆更新期间，主动作进入 loading 或 disabled 状态，禁止重复触发。
- 记忆更新失败时，记忆状态显示 `更新失败`，主动作变为 `重试更新记忆`。

### R7. 章节列表

- 左侧章节列表应保留轻量扫读职责。
- 列表应继续展示写作状态，并仅在有可靠数据时显示轻量记忆状态。
- 列表不提供记忆更新操作。
- 如第一版缺少后端可判定的记忆状态数据，应先保留扩展位或显示写作状态，不臆造状态。

### R8. 第一版记忆状态边界

- 第一版接受“可靠展示优先”：没有后端准确数据时，只显示可由当前页面状态可靠推导的 `不可更新 / 待更新 / 更新中 / 更新失败`。
- `已更新 / 可能过期` 不在第一版中伪造为持久状态；后续需要章节级 memory update 状态或任务摘要接口支持后再补齐。

### Out of Scope

- 不重构批量生成流程。
- 不重构任务中心。
- 不重构生成记录。
- 不重构全局写作页工具栏整体布局。
- 不改变章节状态枚举名称。
- 不删除或回滚已有长期记忆数据。

## Acceptance Criteria

- [ ] 章节详情右上区域分成“章节工具”和“状态工作流”，不再混排所有按钮。
- [ ] `计划中` 章节保存空正文时保持 `计划中`；保存非空正文后自动进入 `草稿`。
- [ ] `计划中` 状态不提供直接定稿主动作。
- [ ] `草稿` 状态有未保存修改时，主动作是 `保存并定稿`，次动作是 `仅保存草稿`。
- [ ] `草稿` 状态无未保存修改时，可手动进入 `定稿`。
- [ ] `定稿` 状态编辑区只读，主动作是记忆更新相关动作，次动作是 `退回草稿`。
- [ ] `定稿 -> 草稿` 需要确认；确认后章节解除只读。
- [ ] 记忆更新只对 `定稿` 章节开放；非定稿章节不能误触发记忆更新。
- [ ] 没有可靠后端数据时，UI 不展示虚假的 `已更新 / 可能过期` 持久记忆状态。
- [ ] 删除入口降级到更多菜单危险项，并保留二次确认。
- [ ] 保存、状态更新、记忆更新期间按钮不可重复点击，并显示处理中反馈。
- [ ] 现有切换章节未保存保护继续生效。
- [ ] 前端状态动作模型有单元测试覆盖 `计划中 / 草稿 / 定稿` 主动作、次动作和禁用规则。
- [ ] 后端保存或状态接口测试覆盖“保存非空正文自动转草稿”的最终接口契约。
- [ ] 前端构建或测试通过；后端相关测试通过。

## Notes
