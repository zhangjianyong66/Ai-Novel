# 前端移动端兼容优化

## Goal

让 Ai-Novel 前端在手机竖屏上完成核心创作闭环，同时保持桌面端工作台效率不退化。

本期移动端范围限定为手机竖屏 360-430px 宽；768px 平板只要求不炸版，不做完整平板重设计。桌面端必须保持现有布局和交互能力。

## Confirmed Facts

- 前端是 React + Vite + TypeScript，主要代码位于 `frontend/src/`。
- 全局布局由 `frontend/src/components/layout/AppShell.tsx` 管理，当前已经存在 `lg:hidden` 的移动端导航抽屉和桌面侧边栏。
- 写作页入口 `frontend/src/pages/WritingPage.tsx` 已拆分为 `WritingWorkspace`、`WritingChapterListDrawer`、`WritingPageOverlays` 等分区。
- 本任务是跨全局布局、通用 UI、核心页面和验证方式的复杂前端兼容改造，需要 `prd.md`、`design.md`、`implement.md` 后再进入实现。

## Requirements

- R1. 设备范围：首批保证手机竖屏 360-430px 宽可用；768px 平板只要求不出现严重炸版；桌面端零回归。
- R2. 首批核心页面：覆盖 `AppShell`、`DashboardPage`、`WritingPage`、`OutlinePage`、`PromptsPage`。
- R3. 全站兜底：非首批页面不要求完整移动端体验，但应避免页面级横向撑爆；复杂表格和调试面板可使用局部横向滚动。
- R4. 导航：移动端保留当前侧边栏信息架构，改造成全屏或近全屏抽屉；不引入底部 Tab。
- R5. 写作页：手机首屏以编辑器主视图为核心；章节列表通过抽屉打开；AI 生成、历史版本、上下文预览、章节分析等继续使用抽屉或弹窗，但必须适配手机高度和安全区。
- R6. 能力分层：一期支持完整创作的最小闭环，包括选章节、编辑正文、保存、AI 生成、章节分析、大纲查看/生成、模型配置保存。
- R7. 降级策略：复杂 diff、RAG 调试、Prompt Studio 高级编辑、结构化记忆图谱、大表格批量编辑可降级为满屏抽屉、局部滚动或基础可操作布局，不追求桌面同等效率。
- R8. 表格策略：Dashboard 项目列表、写作页章节列表、大纲章节列表优先单列化或卡片化；模型配置表单优先单列化，任务预设等复杂表格允许局部滚动；高级调试页先使用滚动兜底。
- R9. 基础设施优先：先修全局容器、导航、通用 `Modal`/`Drawer`、按钮和表单基础样式，再逐页处理核心页面。
- R10. 验证：必须运行前端 lint，并使用移动端和桌面视口检查核心页面；建议加入轻量横向溢出冒烟检查。

## Out of Scope

- 不做原生 App 或 PWA 专项能力。
- 不为所有高级调试页设计完整移动端专属交互。
- 不把所有复杂表格一次性重做成卡片。
- 不引入底部 Tab 导航。
- 不以 768px 平板为首批主要体验目标。

## Acceptance Criteria

- [ ] 360px 宽手机竖屏下，首批核心页面不出现页面级横向滚动。
- [ ] 移动端导航可打开、关闭，点击导航项后自动关闭，不遮挡后续关键操作。
- [ ] 手机竖屏下可以进入项目、选择章节、编辑正文、保存章节、触发章节 AI 生成和章节分析。
- [ ] 手机竖屏下可以查看/生成大纲，并能看到自动保存或失败恢复入口。
- [ ] 手机竖屏下可以完成模型配置的主要表单填写和保存。
- [ ] 通用按钮、表单、抽屉、弹窗在手机宽度下可完整操作，内容不被固定区域遮挡。
- [ ] 复杂列表或表格若无法单列化，必须位于明确的局部滚动容器内，而不是撑出页面级横向滚动。
- [ ] 桌面端 `AppShell` 侧边栏、核心页面布局和主要操作不退化。
- [ ] `cd frontend && npm run lint` 通过。
- [ ] 使用 390x844 和桌面宽度至少检查 `DashboardPage`、`WritingPage`、`OutlinePage`、`PromptsPage`。
